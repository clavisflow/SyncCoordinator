# 関連テーブルを含む同期ルール

## 対応範囲

関連テーブル機能は、同期単位テーブルの1行に対して関連情報を結合して同期する機能です。同期先にProjectionテーブルを追加する必要はありません。

- 同期単位テーブル: 同期先の1行と対応するキーを持つテーブル
- 項目取得: `LEFT JOIN`で関連列をpayloadへ追加
- 対象判定: `EXISTS`で同期対象か判定
- 変更検知: 関連テーブルのTriggerが影響する同期単位キーを列挙して`SyncChangeQueue`へ追加
- 書き戻し: 関連テーブル列は送信元から同期先への片方向のみ

関連情報の読取りはSQL Server、MySQL、PostgreSQLの動的Connectorで行えます。関連テーブル自身の変更検知と、影響する同期単位キーの展開は現在SQL Serverのみ対応します。MySQLまたはPostgreSQLを送信元にする場合は「変更を検知する」を無効にし、同期単位テーブルの変更通知で再読取りできる範囲に限って利用します。

## 受付・作業依頼・スタッフ割当の例

次の構造を想定します。

```text
Reception (受付)
  Id
  ReceptionName

WorkRequest (作業依頼) ← 同期単位
  WorkRequestId
  ReceptionId
  Description

WorkRequestStaff (スタッフ割当)
  WorkRequestId
  StaffNo
```

同期先が受付と作業依頼を非正規化した1テーブルの場合、次のように設定します。

1. 同期単位テーブルに`WorkRequest`を選択する。
2. 項目取得の関連テーブルとして`Reception`を追加する。
3. `{source}.ReceptionId = {related}.Id`を結合式にする。
4. 対象判定の関連テーブルとして`WorkRequestStaff`を追加する。
5. `{source}.WorkRequestId = {related}.WorkRequestId`を結合式にする。
6. 条件式を`{related}.StaffNo IS NOT NULL`にする。`{related}`はその関連テーブル、`{source}`は同期元テーブルとして展開される。
7. `Reception.ReceptionName`などの関連列を同期項目へ追加する。方向は自動的に正方向となる。
8. 作業依頼自身の項目は必要に応じて双方向にする。

受付R1にW1、W2が紐づく場合、Receptionの変更TriggerはW1とW2をそれぞれQueueへ追加します。Workerから先は通常の作業依頼同期として処理されます。

## 対象条件の遷移

`StaffNo IS NOT NULL`を対象条件にした場合の動作は次のとおりです。

| 変化 | 動作 |
|---|---|
| 未登録から登録 | 現在のJOIN結果を同期先へUpsert |
| 登録済みの値を変更 | 現在のJOIN結果を再同期 |
| 登録から未登録 | 対象外としてDelete通知を生成 |
| 初めから未登録の行を追加・変更 | 未同期のため何もしない |
| 作業依頼を変更 | 現在も条件を満たす場合だけUpsert |

対象外のDeleteを同期先へ適用するには、ルールの「削除を同期する」を有効にします。無効の場合は同期先の既存行を維持します。

双方向ルールで削除同期を無効にした場合、対象外になった後も同期先の行が残るため、その行で変更された双方向項目は送信元へ戻り得ます。対象外になった時点で同期関係も終了させる運用では、削除同期を有効にします。

## 項目方向とコンフリクト

- 同期単位テーブルの列は、正方向、逆方向、双方向を選択できます。
- 関連テーブル列は正方向のみです。
- 片方向項目は送信側が正となり、同期先で変更されていてもコンフリクトにせず上書きします。
- 関連テーブル列は逆方向payloadから除外されるため、受付情報やStaffNoを業務DBへ書き戻しません。

## DB配備

関連テーブルを追加・変更するとルールは停止し、DB配備状態がDraftへ戻ります。DB設定画面でSQLを確認し、送信元・同期先へ反映後、検証してルールを再度有効化します。

送信元SQL Serverには次のTriggerが追加されます。

- 同期単位テーブルの通常Trigger
- 「変更を検知する」が有効な関連テーブルごとの展開Trigger

展開TriggerはJOINやpayload生成を行わず、影響する同期単位キーだけをQueueへ追加します。JOINと対象条件の評価はWorkerの読取時に行います。

再配備時は現在の展開Triggerを先に作成または更新し、最後にそのルールの不要な旧Triggerを削除します。関連テーブルの削除、並べ替え、変更検知の無効化を行っても古い展開Triggerは残りません。

## 制約

- 関連テーブルは同期単位テーブルへ直接結合できる必要があります。関連テーブル同士の多段JOINは未対応です。
- 結合式は`{source}`と`{related}`の両方を参照します。関連変更Triggerの性能を保つため、索引のある列同士の等価結合を基本とし、結合列への関数、`CAST`、`OR`などは必要性と実行計画を確認して使用します。
- 項目取得に使う結合は、同期単位1件につき関連行が最大1件になる必要があります。複数行を返した配送はHeldとなり、設定またはデータの修正が必要です。複数行を配列や文字列へ集約する機能は未対応です。
- 対象判定は関連行が複数あっても`EXISTS`として動作します。
- 条件式は関連テーブルごとに1件です。`AND`、`OR`、括弧などを使って複数条件を組み合わせられますが、実際のDBで有効な構文にする必要があります。
- 関連テーブルへの逆方向書込みは未対応です。
- 物理テーブル単位のTrigger共有は未実装で、現在はルール単位でTriggerを配備します。多数のルールを同じテーブルへ設定する場合は事前に負荷検証が必要です。
