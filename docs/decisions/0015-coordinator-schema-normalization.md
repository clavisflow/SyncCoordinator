# ADR 0015: 管理DBのルール構成を現在の集約境界に合わせる

- 状態: Accepted（ADR 0006・0007のschema構成を更新）
- 日付: 2026-07-14

## 決定

- 同期ルールとテーブルマッピングは任意の1対1とし、`RouteTableMapping.RouteId`を主キー兼外部キーにする。
- マッピングの送信先システムは同期ルールから決まるため、`RouteTableMapping`には重複保持しない。
- 列別コンフリクト方針は`RouteColumnMapping.ConflictPolicy`に保持し、独立した`RouteFieldPolicy`は設けない。
- 同期ルールの送信元・送信先はシステムコード文字列ではなく、`SystemDefinition`への外部キーで保持する。
- キュー、Snapshot、競合履歴に含まれるシステムコードは、配送単位や発生時点の記録であるため引き続き保持する。

## 理由

管理画面と実行系はいずれも1ルールにつき1マッピングを前提としている。旧構成ではDBが複数マッピングを許す一方で実行系が`Single`で制約し、列別方針もマッピング保存時に別テーブルへ同期していた。DB制約を実際の仕様へ合わせ、重複値と手続き的な整合性維持をなくす。

## 移行

本決定時点では未運用で既存データの保持が不要なため、段階的な互換移行は作らず、管理DBの初期マイグレーションを再作成する。旧マイグレーションを適用した開発用管理DBは一度削除して再作成する。
