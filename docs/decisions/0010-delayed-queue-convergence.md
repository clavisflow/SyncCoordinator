# ADR 0010: 遅延キューの最新状態への収束

- 状態: Accepted
- 日付: 2026-07-14

## 決定

`SyncChangeQueue`の各行は過去状態を順番に適用する命令ではなく、対象レコードが変更されたことを示す通知として扱う。Workerは取得したbatch内の通知を`EntityType`と`EntityId`でまとめ、各レコードの最新通知だけを処理する。

Connectorの`ReadLatestMessageAsync`は、渡された通知以降にある同一レコードの最新通知を確認し、業務行が存在する場合は現在のpayloadを`Upsert`として返す。業務行が存在せず、最新通知が`Delete`の場合は同じMessageIdの`SyncDeleteTombstone`から削除前payloadを返す。現在状態も有効なTombstoneもない場合は適用対象なしとして返す。

これにより、Workerの停止や障害中に`更新→更新`、`更新→削除`、`削除→再作成`が発生しても、中間状態で送信先を巻き戻さず、処理時点の最新状態へ収束させる。batch取得後に発生した通知はCheckpointより後ろに残るため、次回以降にも再確認される。

Checkpointはbatch内の全レコードについて、関連ルールが完了、保留、適用済み、または適用対象なしになった場合だけbatch末尾まで進める。途中で1件でも失敗した場合は進めず、完了済み配送はInboxと決定的なDeliveryMessageIdによって冪等に再試行する。

競合判定用の`SyncSnapshot`には、送信元で最後に観測したpayloadと、送信先で最後に観測または採用したpayloadを別々に保存する。次回は各側をそれぞれの前回観測値と比較し、以前に送信先を維持した値を送信元の新規変更と誤認しない。削除後は該当側を`null`として記録し、レコードが存在しない状態と空payloadを区別する。

## 適用範囲

この方式は最終状態を一致させるstate synchronizationを対象とする。承認操作、在庫移動、仕訳など中間イベント自体に業務上の意味があるデータは、イベントを省略しない別の連携方式を使用する。

システム単位の一時停止はADR 0011で定める。停止中もTriggerによる通知を保持し、本ADRの最新状態解決を使って再開する。

## 理由

キューへpayload全体を固定保存せず、処理時に業務行を読む構成では、古い通知を変更命令として再生すると最新値の巻き戻しが起こり得る。また、Upsert通知の後に物理削除された行を読むと、後続のDelete通知へ到達する前に処理が停止する。通知を最新状態の再確認トリガーとして扱うことで、通常の遅延、Worker再起動、長時間停止を同じ回復手順で処理できる。
