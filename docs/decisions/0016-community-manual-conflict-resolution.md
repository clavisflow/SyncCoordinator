# ADR 0016: Communityの手動競合解決

- 状態: Accepted
- 日付: 2026-07-16

## 決定

Communityエディションに、個別競合を管理画面から手動解決する機能を含める。

一覧は要対応を初期表示とし、状態ごとに要対応、前の競合待ち、解決処理待ち、解決処理中、解決失敗、解決済み、対応不要（後続優先）、すべてを切り替える。詳細画面では同期先の物理フィールド名を主表示し、受信側フィールド名から同期先フィールド名への対応を補助表示する。項目ごとに受信値、同期先の現在値、手入力値を選択できる。削除競合は削除適用または同期先レコード維持を選択する。業務表示名は必須設定にせず、将来の任意メタデータとして追加できる余地を残す。

解決操作は管理画面から業務DBへ同期適用せず、Coordinator管理DBへ解決要求を永続化する。Workerが要求を取得し、同期先の現在値が画面表示時から変わっていないことを確認して適用する。変化している場合は上書きせず要対応へ戻す。適用には競合IDから導出した決定的Delivery Message IDを使い、業務DBへの適用後にWorkerが中断しても安全に再開できるようにする。

同じ同期経路、同期先、エンティティ、レコードキーで新しい競合が発生した場合は、更新競合と削除競合の組合せを問わずチェーンとして保持する。最古と最新だけを操作可能にし、中間は`WaitingForPrevious`（前の競合待ち）とする。最古から解決した場合は次を更新後の値で再評価し、最新を優先した場合は古い未完了競合を`Superseded`（対応不要（後続優先））にする。Workerの適用中は同じレコードの競合登録と排他する。

競合履歴には発生時点のSnapshot、incoming、current、項目対応、解決要求、理由、要求者、処理状態、失敗内容、解決者、解決日時を保持する。既存の競合履歴は再適用に必要な全payloadを持たないため、migrationでは参照用の解決済み履歴として扱う。

## 理由

競合検出だけでは運用が管理画面内で完結せず、Community版を本番運用できるという製品方針と一致しない。一方、同期先の直接更新は、検出後の変更を上書きする危険と、管理DB・業務DB間の非原子的な失敗を伴う。永続化した要求をWorkerが冪等に適用することで、安全性と再開可能性を確保する。

## 帰結

- 個別の手動解決はCommunityの基本運用機能になる。
- Proは一括解決、自動解決規則、承認フロー、担当者割当等で差別化する。
- 解決処理待ちの間に同期先が変わった場合、運用者は最新値を確認して選択し直す。
- 同じレコードに後続競合が発生した場合、順次解決と最新優先のどちらかを明示的に選べる。
- 物理フィールド名は必ず表示できる。任意の業務表示名を追加しても履歴上の物理名は保持する。
