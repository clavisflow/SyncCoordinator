# ADR 0007: 同期ルールと同期方向

- 状態: Accepted
- 日付: 2026-07-13

## 決定

管理画面では「同期ルート」ではなく「同期ルール」と表現する。1つの同期ルールは固定の送信元、固定の送信先、同期方向（片方向または双方向）、競合設定を持つ。条件による送信先の切り替えは持たせない。

片方向ルールは送信元から送信先だけを処理する。双方向ルールは、送信元から送信されたデータが送信先で更新された場合に限り、最初の送信元へ逆方向に同期する。送信先で独自作成されたデータは双方向ルールの対象にしない。独自作成データを送る必要がある場合は、そのシステムを送信元にした別ルールを作る。

A⇄CとB⇄Cは別ルールとし、A⇄Bは許可しない。CでA由来のデータを更新した場合はAのルールだけ、B由来ならBのルールだけが一致する。

`EntityType`はルールとConnectorを結ぶ内部識別子として残すが、利用者には入力させず、新規ルール作成時に自動生成する。`OriginSystem`も最初の送信元を追跡する内部値として残し、管理画面の設定項目にはしない。

## 理由

利用者が設定したいのは「どこからどこへ、片方向か双方向か」であり、内部のEntity名や発生元による宛先決定ではないため。固定された2地点と明示的な方向にすることで、A由来データがBへ流れる誤設定を避けながら、将来は通常の同期ルール追加でシステムを拡張できる。

## 移行

旧`OriginSystem`ルールが存在する場合、同じEntity Typeでそのシステムを送信先にしていた固定ルールを双方向へ変換し、旧ルールを削除する。既存の内部Entity Typeは保持する。
