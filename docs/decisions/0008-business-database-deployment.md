# ADR 0008: 業務DBの同期オブジェクト反映

- 状態: Accepted
- 日付: 2026-07-13
- 現在仕様: 管理画面認証はADR 0012で追加済み。直接反映は引き続き明示設定が必要で、DBAが確認・実行する方式を本番の標準とする

## 決定

同期ルールとテーブルマッピングの保存では業務DBへDDLを実行しない。新規または構造変更されたルールは下書きとし、専用の「業務DB反映準備」画面で対象DB、変更内容、実行SQL、注意事項を表示する。

作成対象は各業務DBの`SyncChangeQueue`、`SyncAppliedMessage`、`SyncEntityOrigin`、`SyncDeleteTombstone`、`SyncCoordinatorDeployment`と、マッピング対象テーブルの変更検知Triggerとする。既存の業務テーブルや業務列は作成・変更しない。片方向ルールでは送信元だけにTriggerを作成し、双方向ルールでは両側に作成する。

本番の標準手順はSQLをダウンロードし、DBAが確認・実行する方式とする。管理画面からの直接反映は`DatabaseDeployment:AllowDirectApply=true`を明示した環境だけで許可し、SQL確認チェックとDB名の再入力を必須にする。既定値は`false`とする。

実行SQLは最後に`SyncCoordinatorDeployment`へ定義ハッシュを記録する。検証時はテーブルとTriggerの存在だけでなく、このハッシュが現在の設定から生成した値と一致することを確認し、古いSQLを反映済みと誤認しないようにする。

両DBで必要なテーブル、Trigger、定義ハッシュを検証できた場合だけルールを「DB反映済み」にし、その後に有効化できる。DDLに影響するキー列変更は下書きへ戻して無効化する。DB反映済みルールの送信元・送信先・方向・対象テーブル変更は、既存Triggerを残したまま変更できないよう拒否する。

ルールの無効化・削除では業務DBのTriggerを自動削除しない。削除や無効化のDDLは、影響確認を伴う別の廃止フローとして将来追加する。

## 理由

Triggerの追加は業務DBの性能、ロック、既存Trigger、DB権限に影響する。通常の設定保存から分離し、実行SQLを事前確認できるようにすることで、誤ったDBへのDDL実行や無意識の本番変更を防ぐため。

現在の管理画面には認証・認可がないため、直接反映を本番で有効にしてはならない。認証・権限分離、承認者監査を追加するまではDBA実行を本番の必須運用とする。
