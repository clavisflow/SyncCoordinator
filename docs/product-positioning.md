## 1. 一文定義

SyncCoordinatorは、業務アプリケーションごとの連携コードを増やさずに、SQL Server、MySQL、PostgreSQL間の変更を安全に双方向配送し、データ変換、競合、再試行を管理画面で運用できるセルフホスト型のRDB同期基盤です。

### 短縮版

異なる業務データベースを、安全に双方向同期するセルフホスト型コーディネーター。

### 英語版

SyncCoordinator is a self-hosted data synchronization coordinator that safely delivers changes between SQL Server, MySQL, and PostgreSQL, with mapping, conflict resolution, and recovery managed from one operational console.

## 2. 主な対象ユーザー

### 第一対象

複数の業務システムを開発・保守する、企業の情報システム部門および社内開発チーム。

次の条件が重なる組織を主な対象とする。

- 顧客管理、受付、販売、現場作業などの業務システムが分かれている。
- 各システムがSQL Server、MySQL、PostgreSQLのいずれかを使用している。
- 同じ業務データを複数システムへ継続的に反映する必要がある。
- 個別のバッチ、直接DB更新、アプリ間API連携が増え、障害時の追跡や再実行が難しくなっている。
- データと接続情報を自社環境または閉域環境で管理したい。

### 第二対象

- 顧客ごとに異なる業務システムを統合するSIer
- 複数製品間のデータ連携を提供する業務パッケージベンダー
- 段階的なシステム刷新で、新旧システムを一定期間併用する移行チーム

## 3. 解決する課題

複数の業務DBを連携すると、単に値をコピーするだけでは運用できない。

- システムごとにテーブル名、列名、型、最大長、コード体系が異なる。
- 双方向更新では、同じレコードや項目が両側で変更される。
- 一時的な接続障害、Worker停止、処理途中の再起動が発生する。
- 再試行による二重更新と、同期更新を再検知するループを防ぐ必要がある。
- 障害や競合の発生後に、運用者が状況と採用値を確認する場所が必要になる。
- システム追加のたびに個別連携を実装すると、組合せに比例して保守対象が増える。

SyncCoordinatorは、変更配送、マッピング、競合判定、冪等な再試行、運用履歴を共通基盤へ集約する。

## 4. 製品カテゴリー

主カテゴリーは「セルフホスト型RDB間データ同期基盤」とする。

説明の補助として、次の表現を使用できる。

- 業務システム間の変更同期コーディネーター
- 異種RDB間の運用データ同期
- 双方向データ同期と競合管理

「ETL製品」「データウェアハウス連携」「データベースレプリケーション」「ノーコードiPaaS」を主カテゴリーにはしない。これらと一部の機能は重なるが、SyncCoordinatorの中心は、独立した業務システム間で通常業務データを安全に収束させ、運用することである。

## 5. 中心価値

### 異なるDB設計を吸収する

物理テーブルと列を管理画面で対応付け、方向別のコード変換、NULL既定値、UTC正規化、明示的な文字切詰めと数値丸めを設定できる。書込み前に同期先の型、NULL可否、最大長、数値精度を検証する。

### 双方向でも安全に収束させる

項目単位またはレコード単位で競合を判定し、非競合項目は自動マージする。保留した競合は、受信値、同期先の現在値、手入力値を比較して管理画面から解決できる。解決要求はWorkerが現在値を再確認してから冪等に適用する。

### 障害後も処理を継続する

Checkpoint、Inbox、Snapshotを永続化し、一時障害やWorker再起動後も処理を再開する。配送はat-least-onceで行い、決定的なDelivery Message IDと同期先の適用済み記録によって二重適用を防ぐ。

### 運用を一つの画面へ集約する

同期経路、Queue読取位置、Inbox、競合、システムイベント、設定変更、Webhook配送を管理画面から確認できる。システム単位または全体を一時停止し、Workerを再起動せずに接続設定や運用設定を反映できる。

### 自社環境で管理する

Web、Worker、管理DBを自社環境へ配置するセルフホスト構成とする。業務DBの接続情報とWebhook署名秘密鍵は暗号化して管理DBへ保存する。

## 6. 差別化メッセージ

製品ページでは次の順番で訴求する。

1. SQL Server、MySQL、PostgreSQLをまたぐ実際の業務テーブル間同期
2. 列名、型、長さ、ステータスコードが異なるDB間の双方向変換
3. 競合を検出するだけでなく、安全な手動解決まで管理画面内で完結
4. 一時障害、処理途中停止、再起動を前提にした再試行と二重適用防止
5. 変更配送から独立した、署名・再試行・履歴付きWebhook
6. 接続システムや同期ルールをコードへ固定しない構成

## 7. 比較対象

利用者が現在採用している可能性が高い代替手段は次のとおり。

- システムごとの夜間バッチ
- SQLやスクリプトによる直接コピー
- アプリケーション間の個別API連携
- DB製品固有のレプリケーション
- クラウド型iPaaSまたはETLサービス

SyncCoordinatorは、異種RDB上の業務アプリケーションを自社環境で連携し、双方向更新、競合、再試行を運用対象として管理したい場合に適する。

## 8. 非対象と制約

製品ページと動画では、次を実装済み機能として表現しない。

- 既存レコードの初期一括移行
- 送信元と送信先の全件比較および自動修復
- 分析基盤向けの大規模ETL
- データベースのHA、バックアップ、物理レプリケーション
- サブ秒単位を保証するリアルタイム配信
- Exactly-once配送
- SQL Server、MySQL、PostgreSQL以外の標準Connector
- SaaSや外部サービス固有の標準Connector
- 条件による動的な送信先振り分け
- すべての設定を自動化するノーコード導入
- 競合以外の失敗・保留Inboxの手動再実行
- SMTPメール通知、比較・修復、監査エクスポート

既存レコードを同期対象にする場合は、同期用Trigger配備後に業務上の更新通知を発生させるか、別途初期移行手順を用意する必要がある。

## 10. 30秒説明

## 13. 製品ページで最初に示すデモ

Customer Portal、CRM、Field Serviceの3システムを使用し、次の一連の業務を示す。

1. Customer Portalの問い合わせが、列名と型の異なるCRMへ同期される。
2. CRMで作成した作業指示が、ステータスコードを変換してField Serviceへ届く。
3. Field Serviceの作業結果がCRMへ戻る。
4. 同じ問い合わせを両側で変更し、競合を管理画面から解決する。
5. 処理状況画面で配送履歴と再試行可能な構成を確認する。

機能一覧を順番に紹介するのではなく、一つの業務データが3システムを移動し、最後に競合と障害へ対処できることを物語として見せる。
